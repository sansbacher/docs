Basic CLI Info and Guide - for Windows

Many Command Line/CLI guides exist, but often they target macOS or Linux where using Bash in the Terminal is more common. Often the Command Line / CLI / DOS Prompt / PowerShell Console isn't as familiar for Windows Techs or Devs. This is a guide I created for some people at my work a while back, so they could get to grips with the basics of the Command Line on Windows. If you are already familar with navigating around using CLI (Command Line Interface) there's probably nothing new here. These days I would recommend using PowerShell over the traditional CMD/DOS Console - especially since PowerShell has aliases for most DOS/Linux commands so you can just remember one set of commands..

This will mostly be a list of commands/things you should know, and a suggested list of tasks to perform as a kind of "test" (with the results provided so you can check if you're not sure). If you know most of the things here (esp. the commands) you're good to go. If you're not sure then try a CLI Task Test, if you can perform all the tasks then you're good to go (answers are also provided).

If you can't create or change directories, copy/move/rename files using a Command Prompt (PowerShell or otherwise) you should practice a bit; a longer, more complete CLI Crash Course would be Zed's Appendix A of his "Learn Python The Hard Way" (works better in Chrome than IE, or try this link if the first doesn't work). It's designed for both Windows PowerShell, macOS, and Linux and has 15 exercises with practice tasks. Recommended if you've never really used the CLI - it's very much a "type it in and you will learn it" guide, not just a list of commands. You can probably find YouTube Tutorials too. Spend a few lunch hours learning to use the command prompt on your own PC, you won't reget it!

NOTE: This is not a PowerShell or DOS Batch file scripting guide, nor does it delve indepth into using PowerShell (such as variables, objects, the pipeline, etc). It's just for getting around the Command Prompt/CLI. You'll probably learn this as you go (and become more confortable with the CLI).

CLI things you should know:

  • Run powershell.exe to start PowerShell, or look in your Start Menu.
    • Type commands at the PS C:\> prompt and press [Enter] after each.
    • You can generally run any DOS/CMD command (.EXE or .BAT file) in PowerShell, plus newer PowerShell cmdlets, which are usually Verb-Nonu in form, like Get-Content.
    • While you can run regular external command/EXE files, like PING, IPCONFIG, ROUTE, etc from PowerShell sometimes their options/switches cause issues if you need to "enclose things in quotes".
  • Run cmd.exe to start the older "DOS" Command Prompt, or look for "Command Prompt" in your Start menu.
    • Type commands at the C:\> prompt and press [Enter] after each.
    • Generally I would recommend using PowerShell instead, but some commands may work better in the CMD Command Prompt, especially if they have strange switches.
  • PowerShell commands are very verbose/long so most people uses aliases for common commands (eg. using "dir" instead of "Get-ChildItem"). Type alias to see a list of all aliases. The aliases match the old DOS commands, which are shorter, or are abreviations for longer PowerShell commands. There's also aliases for many Linux/Bash/macOS commands.
    • I would only use standard/built-in aliases to ensure they work on systems.
  • There's Tab completion for both commands and paramaters/switches/options, and files/directories. Start typing the command or option and press the [Tab] key to cycle through all choices, which even works for files.
    • For example: Type get-adu<press TAB> and it will complete to Get-ADUser then type -i<press TAB> and it will complete to -Identity after which you can type a username. (assuming you're on a Server 2008 R2 or better Domain Controller)
    • Or type cd \win<press TAB> and it will complete to cd C:\Windows.
    • Press TAB multiple times to cycle through various options, such as Get-AD<TAB several times> will cycle through various AD related commads (eg. Get-ADComputer, Get-ADGroup, Get-ADUser, etc). Works for files / directories too!
  • Paramaters/switches/options in PowerShell are specified after a command with a [Space] and a dash [-], eg. -Identity. You need the space after the command and the dash, you can't type Get-ADUser-Identity (that would be a single long command). Spaces separate options from commands. There's usually also a space after the option if you need to provide a value, like a username or something. But there's no space after the dash, so this wouldn't work: - i
    • Many common switches/options can be abrevated (shorten them enough to make them unique) and some can be skipped if they are implied by being manatory based on their location (eg. you can skip -Identity with Get-ADUser and just type Get-ADUser username as Identity is assumed to be the first paramater).
  • Paramaters/switches/options for DOS/CMD commands (.EXE files) can vary, regardless of using the CMD prompt or PowerShell. The most common cases are:
    • A single dash, like -i, but sometimes a forward shash such as: /i. Sometimes you can stack single character options, such as -abc instead of -a -b -c as separate options
    • Double dashes usually for longer-named-options, such as --do-something
    • And sometimes you just use an option, such as: list
    • A command like git.exe may use all 3:  git log --oneline -n 10
      • 'git' is the command for git.exe. 'log' is a sub-command for git. '--oneline' is a parameter. '-n' is another option with '10' being the value passed to -n.
  • Getting Help: In PowerShell you type help and then cmdlet you want help for, such as: help Get-Process, add -full to see all the details and examples: help Get-Process -full
    • For DOS/CMD/Console .EXE commands it can vary, but one of these should work:
    • -h, /h, --help, /?, -?, or just ?
    • As in ping /?
    • Sometimes you can just type the command with no options and it will provide help, but sometimes the command doesn't require options and it just runs!
      • If this happens and it appears to hang (likely waiting for input) try pressing any of: Ctrl+C, Ctrl+Z, or Ctrl+D to either cancel the command or end the input.
  • The Command Prompt is (generally) NOT Case Sensitive. You can type DIR, dir, DiR, etc. General PowerShell commands are written in CamelCase with words smooshed together using CapitalsToSeparateWords for readability, usually starting with a verb, as in: Get-SomeThingYouWant, or Remove-ThingYouDont.
    • Files are also NOT Case Senitive, though they do retain case (so test.txt = TEST.TXT = Test.TxT, they all reference the same file). Note: UNIX/Linux/mcOS IS Case Sensitive.
    • However sometimes switches/paramters/options for external DOS/.EXE commands are case-sensitive!
  • Because of the need to put "quotes" around commands/files/folders with spaces in them you'll soon discover it's best to avoid spaces in names if possible (or use Tab Completion).
  • To run a script you need to specify the path, if you are in the same directory you still have to specify the current directory, eg: .\test-script.ps1. You can use Tab Completion to start typing the name and press TAB and it will complete the filename along with the ".\" before it.
    • PowerShell scripts are text files that have a .PS1 extension (not .PS, that's PostScript printer files). Windows and most Email/AV/Spam filters will consider them executable so will be blocked from attachments; generally if you want to email a .PS1 file rename it to something like filename-rename-mePS1.txt (but don't use .ps1.txt in case the double extension is caught as suspecious too).
    • DOS/CMD scripts are also text files, but have the extension .BAT (for files containing a batch of commands to run) or .CMD (for CMD, the command processor).
  • Before you can run any script the ExecutionPolicy must be RemoteSigned or Unrestricted. If you get an error running a script type:
    Set-ExecutionPolicy RemoteSigned
  • PowerShell is a Windows program, so be mindful of Run as Administrator, Admin Rights, UAC, etc. If you can't do something, try running your PowerShell prompt as an Administrator.

Using the PowerShell Prompt:

There are several Keyboard Shortcuts that are worth knowing, they make using the CLI much more pleasent and efficient, especially when correcting mistakes or building commands up by adding more options/commands. Most are the same as DOS/CMD. You don't need to memorize them all, but the ones worth learning are:

Up / Down Arrow Scroll Back / Forward through your command history.
Eg. Press UP Arrow to quickly get back to your previous command, press it again to go further back.
Home / End Jump to the Start / End of the command line.
Ctrl+Left / Right Arrow Jump Left / Right one whole word at a time.
(Left / Right arrows otherwise move just one character at a time. And of course Del / Backspace work)
Tab / Shift-Tab Tab completion, as mentioned above. You can also cycle backwards with Shift-Tab.
Esc Clear the current command to start over (there is no undo!)
F7 In DOS/CMD:  Displays a pop-up window of your command history, use arrows and Enter to re-run a command. Doesn't work in PowerShell, type history to see your command history and redo a command with by number with r ##
Ctrl+C Stop/Cancel a long running command that you don't want to wait any longer for. (On older systems Ctrl+Break would exit your whole PowerShell session, but now it appears to do nothing).
Pause Pause a long running command that is scrolling information by too fast.
Press any key to continue. Another option is to pipe (|) it to more to show only one page at a time.
Mouse Right-Click If Quick Edit Mode IS enabled it Pastes from your clipboard. (enabled the same way for PowerShell)
If Quick Edit Mode is NOT enabled it pops up a menu from which you can paste.

Basic File System Commands you should know:

Here's the basic file system commands you should know, these are the aliases for the actual PowerShell commands. I don't know anyone who types Get-ChildItem and while you may see the short alias gci occationally (especially on Microsoft blogs) most people type dir (or ls if they are Linux/UNIX users) because it's what we typed in DOS or the CMD Command Prompt. Most of these you could use in a CMD prompt, so there's a double benefit to learning them!

cls Clears the PowerShell window, short for "clear screen"
dir
 Shows the directory listing (all files / folders), you can specify somewhere else too (eg. dir c:\temp )
md / mkdir
 Make Directory (ie. create a Folder such as: mkdir c:\temp )
cd / chdir
 Change Directory, changes to the specified directory / folder (eg. cd c:\temp )
rd / rmdir
 Remove Directory (if it's not empty it will prompt you if you mean it)
pwd
 Prints the current working directory (which is also shown in your prompt, eg. PS C:\temp>)
copy
 Copy a file, specify source and destination which is often a folder (eg. copy file.txt c:\temp )
move
 Move a file, similar to copy by you don't end up with a second copy - the original is moved.
ren
 Rename a file or folder, specify CurrentName and NewName (eg. ren file.txt "Descriptive Name.txt" )
del
 Delete a file, just specify the file. Does NOT move it to the Recycle Bin and there's no undelete!
write / echo
 Prints something to the screen, if multiple words aren't in "quotes" they will appear on separate lines.
more
 Displays the contents of a text file to the screen, one page at a time if needed. Eg: more file.txt ('q' quits)
help
 Get help for any PowerShell command/alias to see various options and examples (eg. help copy )
alias
 Almost all of these are aliases, alias will show you all aliases, or you can specify one to see the real command.
SOME.EXE
 You can run any regular .EXE file too, eg. ping IP.Add.ress or notepad script.ps1 to edit a script.
exit
 Quits your PowerShell session and closes the window, doesn't ask for confirmation!
.\script.ps1 Runs the named script in the current directory. You can't just type the name, you need to specify the path.

Some Common Console Commands worth knowing:

Here's some common/useful Console commands worth knowing that you may use while at a Command Prompt. These will work in DOS/CMD or PowerShell as they are external commands. You don't need to type them in using ALL CAPS, and you don't need to type .EXE - unless if the name conflicts with a PowerShell alias (eg. SC.EXE is an external command [Service Controller] but SC is a PowerShell alias for Set-Content [write text to a file]. Typing "sc" at a PowerShell prompt is the same as typing Set-Content, but typing "sc.exe" will runn the Service Controller external command). Most of these will display help if you add /? after, as in ping /?

PING and TRACERTSend ICMP Echo packets to check if a host is up or not, eg: ping 1.1.1.1
IPCONFIGCheck your TCP/IP settings to find your IP Address, DNS Server, etc
ROUTEPrint, Add, Change, or Delete routes in your IP route table, eg. route print
NSLOOKUPDNS Name Server Lookup, you can also find Mail (MX), Service (SRV), and Text (TXT) records, eg: nslookup -q=mx gmail.com
DCDIAG and REPADMINRun some Domain Controller (DC) diaganostic checks or check DC Replication state
NET and NETDOMManage local (NET) or AD Domain (NETDOM) users, groups, machines, trusts, and more
NETSHManage and display all sorts of Network Interface/Wireless/Firewall/DNS/DHCP settings
NETSTATDisplay TCP and UDP protocol stats, including what programs have what ports open for connections or IPs with established connections. Eg. netstat -ano
GPUPDATE or GPRESULTForce updating GPOs or check GPO Resultant Set of Policies to see what GPOs are in play
SYSTEMINFO and WHOAMISee information about the System or the currently logged in user, eg: whoami /all
TASKLISTDisplay all the running processes (like Task Manager), you can filter (tasklist /? to see all the various ways) such as: tasklist /fi "imagename eq notepad*"
TASKKILLKill or End a particular task/process, it supports many of the same filtering options as TASKLIST but most offten is by Process ID, add /F to force, eg: taskkill /pid 123456 /f
SC.EXEView, Start, Stop Windows Services - in PowerShell you need to include .exe or it will be an alias of Set-Content. In PowerShell you could also use Get-Service
WHERE.EXEShows the location of a command by searching the PATH environment variable, helpful if the wrong version is being run, eg. where.exe psexec.exe  - in PowerShell you need to incluide .EXE or it will assume you mean Where-Object. In PS you can just use Get-Command
DIR ENV: or SET and PATHShow all environment variables, the first works only in PowerShell, the other two only in CMD.
START and RUNASStart a new process, you can also start a file to open in the associated program, eg to open a Word file use: start "" "some file.docx"  Or start .  to open Windows File Explorer in the current directory. Runas lets you start a process as another user.
NOTEPADNot a command line tool, but if you are editing scripts then notepad scriptFile.ps1 is important.
MSIEXECThe MSI Installer Tool, eg: msiexec /i someProgram.msi /qn /norestart /log logFile.txt
REGView, Set, Change, Import, Export Registry Values or whole Keys.
SHUTDOWNShutdown or Restart the computer, eg: shutdown /r /t 30 you can use /a to abort!
FCFile Compare to see if two files are identical. With text files it shows you the differences.
ATTRIBView, Set or Clear file attrributes, including Read Only, Hidden and System
XCOPY and ROBOCOPYAdvanced Copy commands that can copy while file/folder trees including permissions
GITThe Git source code revision control system command (if installed). Full info on using Git here.
FINDSTR or FINDWhen the output of a command is too long you can | (pipe) it to findstr or find filter based on some text string. Don't use find in PowerShell, in fact in PowerShell you probably want to use Where anyway. But findstr works in both, eg: dir \ | findstr /I "Windows"
SORT and MOREOther commands you can | (pipe) output to, to sort the output or page it one screen at a time - press the [Space] bar for more, eg: dir \windows | more and then press [Space] or "q" to quit.

Important CLI symbols:

There's a few other things you need to know to get around using the Command Line effectively, especially some of the symbols. Again many of these are applicable to the regular CMD Command Prompt as well. You won't need to know/use all of these just now but the more comon ones are near the top.

  .  (dot) The . (dot / period) represents the current directory; thus .\file2.txt means the file2.txt in the directory you are in.
If you are IN c:\temp\folder1\folder2 then . means folder2, and .\file2.txt means c:\temp\folder1\folder2\file2.txt
  ..  (doubledot) The .. (double dots) represents the parent directory (one level up). You can cd .. to move up a directory.
If you are IN c:\temp\folder1\folder2 then .. means folder1, and ..\file1.txt means c:\temp\folder1\file1.txt
  \   (backslash) The \ (backslash) represent the root directory of the drive. It's also used to separate directories and files.
If you are IN c:\temp\folder1\folder2 then \ means c:\, and \file3.txt means c:\file3.txt
  ~   (tilde) In PowerShell the ~ (tilde) represents YOUR home directory, wherever that is. If you are logged in as bsmith then ~\Desktop means c:\users\bsmith\Desktop. It doesn't do anything in CMD/DOS (except for fancy variable stuff)
  #  (hash) Does nothing in DOS/CMD, but in PowerShell the # symbol starts a Comment which is ignored in scripts. In DOS batch file use REM.  <# long comments can cross lines and are enclosed with opening and closing sets of sumbols #> 
  |  (pipe) The | (pipe) is used to send the output of one command as the input of the next command. While CMD supports this PowerShell uses it a lot, eg: dir | select Name, Length | fl  (dir's output goes into select which goes into fl)
  >  (greater) The > (greater than sign) is used to send the output of a comand into a text file, eg. dir > file.txt. This is the "cheap" way of saving output, Use >> (2 greater signs) to append output, as using just one (>) overwrites!
  *  (star)
 The * (or asterisk) is used as a simple wildcard, it represents any number of characters. So dir *cat.txt finds all TXT files that ends in cat (eg. tomcat.txt, nicecat.txt, but not catalog.txt). *cat*.txt would find anything with "cat" in the file name like tomcat2.txt
  ?  (question)
 The ? (question mark) is used as a simple wildcard, it represents any ONE character. Not used as much as *.
But you will see it used as an abreviation for the "where" command in PowerShell.
  %  (percent)
 In DOS/CMD % (percent sign) is used around variables, like %windir%. In PowerShell the % (percent sign) is used as an abreviation for the "foreach" command.
  $  (dollar)
 Not unsed in DOS/CMD, but in PowerShell the $ (dollar sign) is used to start a variable name, so $AllUsers is a variable that might hold "all users", so is $_ which is the implied pipeline variable.
   -  (dash)
 The - (dash) is used to start all options/paramaters/switches on commands. So -Identity denotes the Idenity parameter.
" and ' (quotes)
 The " (double quote) and ' (single quote) are used to enclose strings of characters. My File.txt would be two items, but "My File.txt" is a single item. Use "double quotes" most often, esp. if you want variables to show their values in PS.
   ;  (semicolon)
 The ; (semicolon) is used to separate commands on the same line in PowerShell. So insead of cd \ (press enter) dir (press enter) you can type cd \ ; dir and PowerShell will execute both (change to the root and show the contents of the root).
 & (ampersand)Is the command separator in DOS/CMD (so cd \ & dir would be two separate commands on one line). In PowerShell it invokes/starts a command or script, especially one with a space in the name.
 ` (backtick) or
^ (caret)		These are "escape" characters, which means the shell ignores the next character's special meaning, use the backtick (`) in PowerShell to escape and the caret (^) in CMD/DOS to escape the next character.
 ( ) [ ] { }
 The various brackets: (parentheses), [square brackets], and {curly braces} are used for various things by PowerShell (mainly). The main thing is that they come in matching sets, and are not interchangable. (something] won't work.
